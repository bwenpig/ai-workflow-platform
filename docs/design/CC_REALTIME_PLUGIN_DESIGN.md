# CC 实时推送插件设计

> 文档版本: v1.0 | 日期: 2026-04-08 | 作者: 龙傲天

---

## 1. 背景

### 1.1 当前问题

Claude Code (cc) 作为 OpenClaw 的核心编码代理，在处理复杂任务时存在以下痛点：

- **超时问题**: cc 执行复杂编码任务（如大规模重构、多文件修改）耗时较长，默认 `runTimeoutSeconds` 容易超时导致任务中断
- **无法实时推送**: cc 在后台执行时，用户无法通过微信等即时通讯渠道实时获取进度和结果
- **反馈延迟**: 用户提交任务后只能被动等待，无法知道当前进度，体验差
- **结果丢失风险**: 超时中断可能导致已完成的部分结果无法及时传达

### 1.2 期望目标

- cc 执行长任务时，关键节点实时推送到微信
- 任务完成后自动推送最终结果
- 异常/错误时及时告警
- 用户可在微信端感知任务全生命周期

---

## 2. 可行性分析

### 2.1 subagent 调度 cc

OpenClaw 支持通过 subagent 机制调度 cc：

- `coding-agent` skill 支持以 background process 方式启动 cc
- subagent 可监控 cc 的执行状态和输出
- subagent 完成后结果自动上报给父 agent（push-based completion）
- 支持 `--print` 模式获取 cc 的文本输出

**结论**: ✅ 可行，subagent 是天然的 cc 调度层

### 2.2 runTimeoutSeconds 配置

- 已将 `runTimeoutSeconds` 调整为 3600s（1小时）
- 足以覆盖绝大多数编码任务
- 可根据任务类型动态调整

**结论**: ✅ 超时问题已通过配置解决

### 2.3 openclaw-weixin 消息推送

- openclaw-weixin 插件支持程序化发送消息
- 可通过 channel 机制向指定微信会话推送文本
- 支持 markdown 格式（部分）

**结论**: ✅ 推送通道已具备

### 2.4 技术约束

| 约束项 | 说明 |
|--------|------|
| cc 输出方式 | `--print` 模式输出为文本流，非结构化 |
| subagent 通信 | 仅支持最终结果上报，不支持中间状态流式推送 |
| 微信消息限制 | 单条消息长度有限，需分段发送长内容 |
| 并发限制 | 同一 workspace 下 cc 实例数受限 |

## 3. 方案对比

### 方案 A: 插件 + subagent 编排（轻量方案）

**架构**: 用户 → 微信 → OpenClaw 主 agent → spawn subagent(cc) → cc 执行 → subagent 结果回传 → 主 agent 推送微信

**流程**:
1. 主 agent 收到任务，spawn coding subagent
2. subagent 调用 cc 执行编码任务
3. cc 完成后 subagent 返回结果
4. 主 agent 收到结果后通过 weixin channel 推送

### 方案 B: cc CLI + weixin API 直接集成（自研方案）

**架构**: 自研 wrapper 脚本 → 启动 cc CLI → 解析输出流 → 调用 weixin API 实时推送

**流程**:
1. wrapper 脚本接收任务参数
2. 启动 cc 进程，pipe stdout/stderr
3. 解析输出，识别关键节点（文件修改、测试结果等）
4. 通过 weixin HTTP API 逐条推送

### 方案 C: ClawFlow 工作流编排（重量方案）

**架构**: 用户 → 微信 → OpenClaw → ClawFlow workflow → 多阶段 task → 各阶段推送

**流程**:
1. 定义 ClawFlow workflow，包含：任务分解 → 逐步执行 → 结果汇总
2. 每个 task 完成后触发推送 hook
3. 支持条件分支（成功/失败走不同路径）
4. 最终汇总推送

### 方案对比表

| 维度 | A: 插件+subagent | B: cc CLI+weixin | C: ClawFlow 编排 |
|------|------------------|-------------------|-------------------|
| **开发量** | 低（利用现有能力） | 中（需开发 wrapper） | 高（需设计 workflow） |
| **实时性** | ⚠️ 仅最终结果推送 | ✅ 可逐行/逐步推送 | ✅ 阶段性推送 |
| **可靠性** | ✅ 依赖成熟的 subagent 机制 | ⚠️ 需自行处理进程管理 | ✅ ClawFlow 内置重试 |
| **灵活性** | ⚠️ 定制空间有限 | ✅ 完全可控 | ✅ 高度可编排 |
| **维护成本** | 低 | 中 | 高 |
| **扩展性** | ⚠️ 难以添加中间推送 | ✅ 可随意扩展推送点 | ✅ 天然支持多阶段 |
| **上手时间** | < 1天 | 2-3天 | 3-5天 |

## 4. 推荐方案

### 推荐: 方案 A 为基础 + 方案 B 的关键特性 → 混合方案 A+

**理由**:

1. **快速落地**: 方案 A 利用现有 subagent 机制，几乎零开发即可跑通基本流程
2. **渐进增强**: 在方案 A 基础上，增加一个轻量 wrapper 层实现关键节点推送（借鉴方案 B 思路）
3. **避免过度设计**: 方案 C 虽然强大，但当前需求不需要复杂工作流编排
4. **实时性补偿**: 通过将大任务拆分为多个小 subagent 串行执行，每个完成后立即推送，近似实现实时效果

### 混合方案 A+ 架构

```
用户(微信) → OpenClaw 主 agent
                ↓
         任务分析 & 拆解
                ↓
    ┌──── spawn subagent-1 (分析阶段) ────┐
    │     cc: 分析代码/理解需求            │
    │     完成 → 推送"分析完成"到微信      │
    └──────────────────────────────────────┘
                ↓
    ┌──── spawn subagent-2 (实现阶段) ────┐
    │     cc: 编写代码/修改文件            │
    │     完成 → 推送"实现完成"到微信      │
    └──────────────────────────────────────┘
                ↓
    ┌──── spawn subagent-3 (验证阶段) ────┐
    │     cc: 运行测试/检查结果            │
    │     完成 → 推送"验证完成"到微信      │
    └──────────────────────────────────────┘
                ↓
         汇总结果 → 推送最终报告到微信
```

### 核心设计要点

1. **任务拆分粒度**: 将大任务拆为 2-4 个阶段性 subagent，每阶段完成后推送
2. **推送内容模板**: 统一格式，包含阶段名、状态、关键变更摘要
3. **异常处理**: subagent 失败时立即推送错误信息，不静默丢失
4. **超时保护**: 每个 subagent 独立超时（默认 900s），避免单点卡死

## 5. 实现步骤

### Phase 1: 基础能力验证（0.5 天）

1. **验证 subagent 调度 cc 的完整链路**
   - 手动 spawn 一个 coding subagent 执行简单任务
   - 确认结果能正确回传到主 agent
   - 确认主 agent 能通过 weixin channel 推送结果

2. **确认 runTimeoutSeconds 配置生效**
   - 执行一个耗时较长的 cc 任务（>60s）
   - 验证不会因超时中断

### Phase 2: 任务拆分框架（1 天）

3. **设计任务拆分策略**
   - 定义标准阶段模板：`分析 → 实现 → 验证`
   - 编写 prompt 模板，指导主 agent 如何拆分任务

4. **实现推送消息模板**
   ```
   🔧 [任务名称] - 阶段 N/M
   状态: ✅ 完成 / ❌ 失败 / ⏳ 进行中
   耗时: Xs
   摘要: ...
   ```

### Phase 3: 插件封装（1 天）

5. **封装为 OpenClaw Skill**
   - 创建 `cc-realtime` skill 目录
   - 编写 SKILL.md，定义触发条件和执行流程
   - 包含任务拆分逻辑、推送模板、错误处理

6. **配置项**
   - 推送目标 channel（默认 openclaw-weixin）
   - 阶段超时时间
   - 推送详细程度（简洁/详细）
   - 是否推送中间过程

### Phase 4: 测试 & 优化（0.5 天）

7. **端到端测试**
   - 微信发送编码任务 → 收到阶段推送 → 收到最终结果
   - 测试超时场景
   - 测试 cc 执行失败场景

8. **优化推送体验**
   - 调整推送频率，避免消息轰炸
   - 优化摘要生成质量
   - 添加任务取消能力（可选）
